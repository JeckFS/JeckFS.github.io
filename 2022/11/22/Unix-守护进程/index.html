Unix-守护进程

发表于2022-11-22|更新于2022-11-24|Unix
|阅读量:

守护进程的特征

  • 内核级守护进程
    • 父进程 ID 为 0 的进程通常是内核进程，它们作为系统引导装入过程的一部分而启动。
    • 通常存在于系统的整个生命期中，它们以超级用户特权运行，无控制终端（终端名设置为问号），无命令行
    • 在 ps fajex 的输出实例中，内核守护进程的名字会用方括号包裹。
  • 用户级守护进程
    • 大多数用户层守护进程都是进程组的组长进程以及会话首进程，而且是这些进程组和会话中的唯一进程（rsyslogd 除外）
    • 父进程是 init 进程
    • 无控制终端（终端名为问号，可能是调用了 setsid()），无命令行

内核级守护进程举例：

  • kthreadd：用来创建其他内核进程，是其他内核进程的父进程。

编程规则

  1. 调用 unask 函数将文件模式创建屏蔽字设置为一个已知值，通常为 0.
    原因是：一是继承的文件模式创建屏蔽字在守护进程创建文件时只能按照屏蔽字去设置权限；二是在使用库函数创建文件时由于不能显式指定文件权限，使用 umask 设置一个具体值会更有意义。

  2. 子进程调用 fork ，并使父进程调用 exit。
    这实现了两点：

    • 若守护进程以一条简单的 shell 命令启动，则父进程终止会让 shell 认为这条 shell 命令执行完毕。
    • 子进程继承了父进程的 group ID，但子进程 ID 却与父进程不同，这保证了子进程不是进程组的组长。

    这一步也是下一步调用 setsid 的先决条件。

  3. 调用 setsid 创建新会话
    分三步实现：

    • 让进程先变为新会话的首进程
    • 然后变为新进程组的组长进程
    • 最后解除控制终端
  4. 将当前工作目录变为 root 目录。
    子进程的工作目录继承自父进程，该目录是一个已挂载的文件系统。由于守护进程随着系统的运行而运行，因此若不改变工作目录则父进程的工作目录永远不能被解挂。
    也可以将工作目录设定为一个具体的目录，并不一定是 root 目录。
  5. 关闭不需要的文件描述符。
    这些描述符有可能是从父进程、shell 进程或其他进程继承来的。可以使用 open_max() / getrlimit() 获取最大的文件描述符值并关闭所有的小于等于该值的文件描述符。
  6. 某些守护进程将标准输入、标准输出、错误输出重定向到 /dev/null 中以达到禁用的效果。

守护线程打印错误日志

守护线程由于没有控制终端因此并不能使用标准的错误输出。大多数守护进程是通过 BSD syslog 设备进行错误输出。

生成日志信息的三种方法：

  • 内核例程调用 log()。
    任何用户进程都可以通过打开并读取 /dev/klog 设备来读取这些日志。
  • 大多数用户级守护进程调用 syslog 生成日志信息。
    这类信息被发送至 /dev/log
  • 通过 UDP 514 端口接收本机或远程主机的日志。
    syslog() 并不生成 UDP 数据报，而是需要该守护进程通过网络编程自己发送。

通常，syslogd 守护进程会读取上述三种日志。它在启动时会读取 /etc/syslog.conf 文件，该文件决定了不同种类的日志被发送到何处。如紧急信息会被送往系统管理员（若登录）并在控制台打印，但是警告信息会被记录到文件中。

  • openlog() 是可选的。若不调用 openlog，当第一次调用 syslog 时，openlog 自动被调用。

  • closelog() 是可选的。作用是关闭用来与 syslogd 守护进程通信的描述符

  • syslog() 用于生成一条日志信息
  • setlogmask() 用于设置进程的日志优先级屏蔽字，并返回原先的屏蔽字。设置日志优先级屏蔽字后，只有日志优先级屏蔽字中包含的该日志信息的优先级，该日志才可以被记录。maskpri = 0 时 setlogmask 将不起任何作用。

补充教程

单实例守护进程

单实例指的是一些守护进程为保证操作的正确性，在同一时间只能有一个该进程副本运行。

应用场景 / 应用原因：

  • 守护进程需要对设备进行专有名词的访问。
    如 cron 单例守护进程，若存在多个会导致一个调度任务被启动多次的问题。
  • 设备驱动有时会阻止 /dev 下设备节点的多次访问尝试。因此多守护进程对同一个设备的多次访问有时会失败。
  • 文件锁和记录锁机制的存在保证守护进程仅有一个副本在运行。
    每个守护进程的副本创建一个固定名称的文件并对整个文件上加写锁，会导致其他守护进程无法创建写锁。（写锁会在守护进程退出时自动解锁）

守护进程的惯例

  • 惯例一：若守护进程使用锁文件，则该文件需要存储在 /var/run (需有超级用户权限)中。命名规则：[daemon / service name].pid
  • 惯例二：若守护进程支持配置，则配置文件应存储在 /etc 中，命名规则：[daemon / service name].conf
  • 惯例三：守护进程可以以命令行启动，但通常从系统初始化脚本（/etc/rc 或 /etc/init.d/）中启动。若在 /etc/inittab 中包含该守护进程的 respawn 记录项，当该守护进程退出后，init 进程会重启该守护进程。
  • 惯例四：一般守护进程在启动时读取配置文件，通常不会再次读取它，即使发生更改。若想使配置生效，一种是重启该守护进程，另一种是守护进程通过捕捉 SIGHUP 信号来读取配置。因为守护进程要么是无控制终端的会话首进程，要么是孤儿进程组的一员，因此 SIGHUP 信号对守护进程来说是可重复使用的，并不会产生歧义。

客户端-服务器模型

守护进程的一个典型应用是作为服务进程，如 syslogd 就是一个服务端的角色。

每个服务进程（守护进程）若不设置就会继承父进程的文件描述符，包括访问端点、配置文件、日志文件等。这会引起一些安全问题，如守护进程修改了配置文件、欺骗客户端认为正在与服务端通信等。一个可行的方案是将守护进程不需要的文件描述符关闭：

FD_CLOEXEC：在调用 exec 系列函数后，该文件自动关闭。
