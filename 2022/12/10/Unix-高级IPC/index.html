Unix-高级IPC

发表于2022-12-10|更新于2022-12-11|Unix
|阅读量:

高级 IPC 通信的形式：the UNIX domain socket mechanism（与此相对应的是 Internet domain socket）。
运行在同一个计算机系统中的不同进程可以互相传递已打开的文件描述符。服务端进程为其文件描述符命名，客户端进程（与服务端进程运行在同一个系统）可以使用该名字与服务端通信。

UNIX domain socket

UNIX domain socket 与 Internet domain socket 不同，它无协议处理程序，无网络首部数据的添加或删除，无校验码的计算，无序列号的生成，无确认信息的发送。

UNIX domain socket 提供了基于 Stream 和 datagram 两种形式的接口。datagram 形式的 UNIX domain socket 能够提供可靠的服务，多个信息几不会丢失也不会乱序到达。

创建一对未命名的，已经建立连接的 UNIX domain socket。

相互连接的成对 UNIX domain socket 就像一个全双工的管道，两端都是允许读写的。这里将其称为 fd-pipes 以显式与普通半双工管道的区别。

由于 XSI 消息队列并不与文件描述符相关联，因此它无法使用 poll 或 select 函数。这里借助 UNIX domain socket XSI 消息队列可以间接使用 poll 或 select 函数。

Naming UNIX Domain Sockets

通过 socketpair 函数创建的 Socket 虽然彼此连接但属于未命名，仅能在这两个 Socket 之间进行通讯。使用 Naming UNIX Domain Socket 允许不相关的进程进行通讯。

UNIX domain socket 的地址使用数据结构 sockaddr_un 来进行表示：

当将一个地址绑定到 UNIX domain socket 上是，系统会创建一个名为 sun_path 且类型是 S_IFSOCK 的文件。该文件的存在仅仅作为将 Socket name 告知给客户端的一种方式。该文件用来与通讯，不能被 open 操作。
若绑定到同一个地址时该文件已经存在，bind 函数将会失败。
关闭该 Socket 时，该文件也不会被自动删除，因此在程序推出之前应该对该文件调用 unlink 函数，使之成为临时文件。

唯一连接（Unique Connections）

服务端使用标准的 bind()、listen()、accept() 与客户端建立唯一的 UNIX domain 连接。客户端调用 connect() 成功后，客户端和服务端的唯一连接就建立起来了。

上述三个函数用于同一个机器中的不相关的进程之间建立唯一连接。这些函数模仿了面向连接的 Socket 函数。这里的底层通讯机制是基于 UNIX domain socket 的。

传送文件描述符

传递成功后，客户端与服务端共用同一个 file table entry（如图 17.11），然后关闭发送端的文件描述符（接收端的仍然打开）。

文件描述符传送函数：

通常是服务进程调用 send_fd 或者 send_err 函数将一个描述符传递给客户进程。客户进程通过调用 recv_fd 接收该文件描述符。

  • send_fd：将文件描述符 fd_to_send 通过 UNIX domain socket（由参数 fd 表示）。
  • send_err：将 errmsg 通过 UNIX domain socket（由参数 fd 表示) 发送。参数 status 值域为 [-255, -1]
  • recv_fd：调用成功返回 send_fd 传送的文件描述符，否则返回由函数 send_err 指定的 status 参数的值。若服务进程发送了一条错误信息，客户进程的 userfunc 函数将被调用来处理该出错信息。userfunc 函数的第一个参数为 STDERR_FILENO 常量；第二个参数为指向错误信息的指针；第三个参数为错误信息的长度。
    通常将 write 函数传递给 userfunc 参数。

send_err 的实现

  • send_fd：先发送了 2 个 0 字节，然后紧随其后的是实际的描述符。
  • send_err：先发送 errmsg ，然后发送一字节的 0，最后发送 status 的绝对值。
  • recv_fd：接收 socket 上的所有内容直到遇到了 NULL 字节。接收的所有内容都传送给 userfunc 函数。然后 recv_fd 读取到的 status 的绝对值。若 status 为 0，则描述符被成功传送，否则无描述符可接收。

通过调用 sendmsg 和 recvmsg 函数实现交换文件描述符的功能。这两个函数都有一个指向 msghdr 结构的指针，该指针包含了发送 / 接收的所有信息。

  • 属性 msg_name 与 msg_namelen：基于网络连接发送 datagram 时，通常使用这两个参数
  • msg_iov 与 msg_iovlen：指定缓冲数组（用于离散读或集中写，scatter read or gather write，见 Section 14.6 readv / writev）
  • msg_flags：包含了描述接收的消息的标志，取值如下：
  • msg_control 与 msg_controllen：用于处理传递货接收控制信息。msg_control 指向 cmsghdr 结构（control message header）；msg_controllen 表示控制信息的字节长度。
    • cmsg_len：cmsghdr 结构体的长度 + 描述符的长度
    • cmsg_level：设置为 SOL_SOCKET
    • cmsg_type：设置为 SCM_RIGHTS（SCM：socket-level control message），表示传送访问权限（访问权限仅能通过 UNIX domain socket 来传送）。
    • cmsg_data：位于 cmsg_type 属性之后，用于存储指向描述符的指针。

  • CMSG_LEN 宏返回存储 nbytes 长的数据对象所需的字节数，它先将 nbytes 加上 cmsghdr 结构的长度，然后按处理器体系结构的对齐要求进行调整，然后再向上取整。

在 serv_accept 中，获取调用者的 credentials 的最好的办法是在调用 accept 函数时由内核将调用者的 credentials 传递给我们。但一些 UNIX domain socket 在实现相似的功能时通常差异较大。如在 FreeBSD 8.0 和 Linux 3.2.0 中提供了通过 UNIX domain socket 发送 credentials 的功能（两者的使用方式也不一样）；Mac OS X 10.6.8 中部分继承自 FreeBSD，但仅用了传送 credentials 的功能；Solaris 10 根本不支持通过 UNIX domain socket 传送 credentials，但它通过 STREAMS 管道传送文件描述符的方式提供了获取一个进程的 credentials 的功能。
