1.容器

1.1 组件添加

1.1.1 @ComponentScan

@ComponentScan的常用属性:

  • value属性指定要扫描的包.
  • excludeFilters:指定排除扫描哪些注解
  • includeFilters:指定只包含哪些注解.
  • userDefaultFilters=false属性与includeFilters属性一起使用.

@ComponentScan其他特性:

  • 重复使用

1.1.1.1 @Filter注解

常用属性:

  • type：指定过滤的类型
    FilterType.ANNOTATION：按照注解形式过滤

    FilterType.ASSIGNABLE_TYPE：按照给定的类型进行扫描

    FilterType.ASPECTJ：使用ASPECTJ表达式
    FilterType.REGEX：使用正则指定
    FilterType.CUSTOM：使用自定义规则

  • classes：过滤的对象

1.1.2 @Scope

@Scope取值：

  • prototype：多实例，当使用该对象时，才会创建对象，且不会放入ioc容器中。每次获取该对象都创建新的对象。
  • singleton：单实例(默认值)，ioc容器启动后创建对象，并将该对象放入ioc容器中。
  • request：同一次请求创建一个实例
  • session：同一个session创建一个实例

1.1.3 @Lazy

懒加载只能用来单实例bean中。设置该bean在第一次使用时再创建该对象。

1.1.4 @Conditional ☆☆☆

按照一定的条件进行判断，满足条件给容器中注册bean。

注册在方法上：只有满足条件才会注册该bean
注册在配置类上：只有满足条件，该配置类才会生效，其下的bean才会被注册

需求：有两个bean，一个是“bill”，一个是“linus”。在windows环境下加载bill bean，在Linux环境下，加载linus bean。


1.1.5 @Import ☆☆☆

第一种导入方式

第二种导入方式：使用ImportSelector导入。
实现ImportSelector接口，并重写selectImports(AnnotationMetadata)，该方法返回的String数组包含了需要导入的bean的全类名。


第三种导入方式：使用ImportBeanDefinitionRegistrar

业务逻辑：当ioc中已经注册了com.atguigu.bean.Red和com.atguigu.bean.Blue，才注册RainBow这个bean。

1.1.6 FactoryBean

使用FactoryBean默认只能获取到FactoryBean中getObject()中返回的类型。若想获取FactoryBean需要加入”&”前缀来获取。
Object bean4 = applicationContext.getBean("&colorFactoryBean");

小结：在IOC中注册bean的几种方法：

  • 使用@Bean
  • 使用@Import
    • 根据bean的类型直接导入
    • 使用ImportSelector接口导入
    • 使用ImportBeanDefinitionRegistrar接口导入
  • 使用FactoryBean导入

1.2 Bean的生命周期

方式一：使用@Bean的initMethod,destroyMethod属性指定Bean创建前后的操作

单实例Bean在容器关闭时销毁；
多实例Bean容器不会管理这个Bean，更不会调用这个对象的销毁方法。

方式二：使用InitializingBean接口初始化，DisposableBean接口销毁

方式三：使用JSR250的@PostConstruct，@PreDestroy注解

使用@PostConstruct注解修饰的方法在调用该方法所在类的构造函数调用完成之后，才会被调用。

方式四：使用BeanPostProcessor

postProcessBeforeInitialization()与postProcessAfterInitialization()的调用时机：

  1. 先执行该bean对象的构造方法
  2. 执行postProcessBeforeInitialization()
  3. 执行init-method, InitializingBean的afterPropertiesSet()，@PostConstruct修饰的方法
  4. 执行postProcessAfterInitialization()

BeanPostProcessor的原理：

由于会定义多个BeanPostProcessor，他们在执行的时候是顺序执行的，当某个BeanPostProcessor的postProcessBeforeInitialization()或postProcessAfterInitialization()返回为null后，后续的BeanPostProcessor也不会执行。源码如下：

  1. 通过populateBean()方法对bean对象，不是BeanDefinition进行属性赋值
  2. 调用initializeBean()对bean进行初始化，初始化前后调用BeanPostProcessor中的方法
    在initializeBean()中进行后置处理器的调用
    1. 通过applyBeanPostProcessorBeforeInitialization()执行所有BeanPostProcessor的postProcessBeforeInitialization()
    2. 再通过invokeInitMethods()执行init-method, InitializingBean的afterPropertiesSet()，@PostConstruct修饰的方法
    3. 通过applyBeanPostProcessorAfterInitialization()执行所有BeanPostProcessor的postProcessAfterInitialization()

BeanPostProcessor在spring底层的应用：

1. ApplicationContextAware

通过ApplicationContextAware获取ApplicationContext

ApplicationContextAware是通过BeanPostProcessor的方式获取到ApplicationContext的。

2.BeanValidationPostProcessor用于bean的校验
3.InitDestroyAnnotationBeanPostProcessor用于实现JSR250中的@PostConstruct，@PreDestroy
4.AutowiredAnnotationBeanPostProcessor实现@Autowired注解
5.AsyncAnnotationBeanPostProcessor实现@Async注解的实现

1.3 组件赋值/自动装配

1.3.1 @Value、@PropertySource与@PropertySources

@Value给bean中的基本类型注入值。支持SpEL与${}
@PropertySource指定properties文件的位置。
@PropertySources指定多个@PropertySource。

通过环境变量获取properties配置文件中的值

1.3.2 @Autowired、@Qualifier与@Primary

@Autowired：
默认按照类型(byType)去容器中找对应的组件找到就赋值；
如果找到多个相同类型的组件，再将属性的名称作为组件的id去容器中查找。

@Qualifier：
存在同一个类型被注册进多个对象，@Qualifier注入具体id的对象。

@Primary：当IOC容器中，同一个类型存在多个Bean，优先加载@Primary修饰的Bean。若@Qualifier指定了具体的Bean则@Primary失效。

@Autowired修饰方法、修饰构造方法、修饰方法参数。需要的组件都是从容器中获得。
若只有一个有参构造器，@Autowired可以省略；
配置类中的方法参数可以省略@Autowired，可以自动完成注入。

1.3.3 @Resource(JSR250)与@Inject(JSR330)

@Resource默认按照组件的属性名称进行装配。也可通过其name属性指定具体装配的bean的id。相比与@Autowired，它不能与@Qualifier、@Primary配合使用，也没有@Autowired(require=false)的功能。

@Inject的使用需要导入javax.inject的依赖。Inject没有@Autowired(required=false)的功能。

1.3.4 使用xxxAware接口获取spring运行过程中的组件

xxxAware的功能大都是(不完全是)依靠对应的xxxAwareProcessor来实现的。

1.3.5 @Profile

@Profile：指定组件在哪个环境中才能被注册到容器中。加了环境标识的bean只有在这个环境被激活时才能被加入到IOC容器中。默认是default环境。
@Profile可以修饰配置类中的方法，也可以修饰整个配置类。
没有标注@Profile的组件，在任何环境下都是可以被加载的。

示例：在测试环境、开发环境、默认环境中使用不同的数据源。

环境激活：
方式一：使用命令行参数：-Dspring.profiles.active=test
在Run Configurations中配置VM arguments： -Dspring.profiles.active=test，指定当前的环境是test环境。当不指定时，使用的是default环境。

2. 扩展原理

2.1 BeanFactoryPostProcessor

All bean definitions will have been loaded, but no beans will have been instantiated yet. This allows for overriding or adding properties even to eager-initializing beans.

2.2 BeanDefinitionRegistryPostProcessor

BeanDefinitionRegistryPostProcessor继承自BeanFactoryPostProcessor

【执行时机】：
All regular bean definitions will have been loaded, but no beans will have been instantiated yet.

BeanDefinitionRegistryPostProcessor先于BeanFactoryPostProcessor执行。
先执行BeanDefinitionRegistryPostProcessor.postProcessBeanDefinitionRegistry()
—> BeanDefinitionRegistryPostProcessor.postProcessBeanFactory()
—> BeanFactoryPostProcessor.postProcessBeanFactory()

2.3 ApplicationListener

监听容器中发布的事件。事件驱动模型开发；
public interface ApplicationListener<E extends ApplicationEvent>
监听 ApplicationEvent 及其下面的子事件；

ContextRefreshedEvent：容器刷新完成（所有bean都完全创建）会发布这个事件；
ContextClosedEvent：关闭容器会发布这个事件；
