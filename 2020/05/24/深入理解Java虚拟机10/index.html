  • [前端编译与优化]
    • 概述
      • 前端编译器
        • 将＊.java文件转变成＊.class文件
        • 如：Javac，Eclipse JDT中的增量式编译器(ECJ)
        • 支撑着程序员的编码效率
      • 即时编译器(JIT编译器)
        • 在运行期把字节码转变成本地机器码
        • 如：HotSpot中的C1，C2编译器，以及Graal编译器
        • 决定了程序的执行效率。
      • 静态的提前编译器(AOT编译器)
        • 直接把程序编译成与目标机器指令集相关的二进制代码
        • 如：jdk的Jaotc
    • Javac编译器(用Java实现)
      • 编译过程
        • 1.准备过程：初始化插入式注解处理器
        • 2.解析与填充符号表过程
          • 2.1.词法分析：将源代码的字符流转化为标记集合的过程。
            • 标记是编译时的最小元素。
            • int a=b+2;中的标记有：int,a,=,b,+,2.
          • 2.2.语法分析：根据标记序列构造抽象语法树(AST)
            • 抽象语法树的每个节点代表程序代码中的一个语法结构。如包，类型，修饰符，运算符，接口，返回值，代码注释等。
            • 抽象语法树都是基于抽象语法树的，而不是源码字符流。
          • 2.3.填充符号表。产生符号地址和符号信息。
            • 符号表是一组符号地址和符号信息构成的数据结构。类似哈希表中的键值对的存储形式。
        • 3.插入式注解处理器的注解处理过程。见实战。
          • 整体认识
            • 在设计上原本是与普通Java代码一样，在程序运行期间发挥作用的。
            • 使用“插入式注解处理器”的标准API，提前至编译期对代码中的特定注解进行处理。
            • 该API允许在处理注解期间读取，修改，添加抽象语法树中的任意元素。若AST发生修改，则返回到解析及填充符号表的过程重新处理。这一循环过程称为一个“轮次(Round)”。
          • 应用：Lombok，通过注解生成getter/setter方法，产生equals()与hashCode()方法等。
        • 4.语义分析与字节码生成过程
          • 语义分析
            • 标注检查。
              • 用途：变量使用前是否被声明、变量与赋值之间的数据类型是否匹配等。
              • 优化技术：常量折叠。
            • 数据流及控制流分析。对程序动态运行过程进行检查。
              • 用途：程序局部变量在使用前是否赋值、方法的每条路径是否都有返回值、是否所有的受查异常都被正确处理等问题。
              • 应用：final语义校验。
          • 字节码生产过程
            • 解语法糖。将简化代码编写的语法糖转化为原有的形式。
              • 对编译结果和功能无影响。可以减少代码量。Java虚拟机不支持语法糖，语法糖在编译阶段被还原回原始的基础语法结构。
              • 应用：
                • 泛型
                • 变长参数
                • 自动装箱拆箱
            • 字节码生成
              • 将语法树、符号表等前面步骤生成的信息转化为字节码指令存储到磁盘。另外将实例构造器<init>()方法与类构造器<clinit>()添加到语法树中。

常量折叠：
int a = 1 + 2;

final语义校验：
编译后的字节码是一模一样的。final变量的不变性仅由javac编译器在编译期间保证。

<init>()与<clinit>()的作用：
static{}块、变量初始化（实例变量和类变量）、调用父类的实例构造器等操作收敛到<init>()与<clinit>()中执行。

Java语法糖

1.泛型

1.1 实现泛型的方法

  1. 类型擦除式泛型
    只存在于源码中，在编译后的字节码中，全部泛型都被替换成原来的裸类型。并在相应地方插入了强制类型转换代码。

    ArrayList<int>与ArrayList<String>是同一个类型.

    缺点:
    a) 无法使用泛型创建对象和数组. 即运行期无法获取泛型类型信息.

    b) 不支持原生类型. 在实现泛型过程中需不断地拆箱装箱, 耗费性能.
    因为Java在类型擦除后所有的类型都转化为Object, 而原生类型如int, double等不能与Object类型转换, 所以int要转化为Integer, double要转化为Double. 在存取数据时原生数据需要不断的自动装箱和拆箱. 可参考类型擦除.

    c) 泛型参数的重载缺陷
    虽然重载要求方法的特征签名(不包括返回值)不同, 但是在参数是泛型时有些特殊, 返回值的不同可以标识不同的函数.

    在49.0以上版本的class文件的虚拟机可以识别Signature参数. 为解决获取泛型类中的参数化类型,引入了Signature, LocalVariableTypeTable属性.
    Signature属性可以存储一个方法在字节码层面的特征签名, 其保存了参数化类型的信息,所以可以在运行时获取泛型的参数化类型. 这在虚拟机解析,反射等情况下要用到.
    从Signature属性的出现我们还可以得出结论，擦除法所谓的擦除，仅仅是对方法的Code属性中的字节码进行擦除，实际上元数据中还是保留了泛型信息，这也是我们在编码时能通过反射手段取得参数化类型的根本依据。
    [ps: 一个Java程序中的信息分为代码（Code，方法体里面的Java代码）和元数据（Metadata，包括类、字段、方法定义及其他信息）两部分。]

    优点:
    a) 不需改动字节码和Java虚拟机, 可保证老版本中没有使用泛型的代码仍可运行在后续支持泛型的代码中. 即向前兼容.

    在源码中,编译后的中间语言表示上, 运行期的CLR中都是切实存在的.List<int>与List<string>是不同的类型,它们由系统在运行期生成.

2.自动装箱,拆箱,遍历循环与变长参数

自动装箱,拆箱 —-> Integer.valueOf(), Integer.intValue()
遍历循环 —-> 迭代器实现 (被遍历的类需要实现Iterable接口)
变长参数 —-> 变为数组类型参数

自动装箱的陷阱:

  • 包装类的“==”运算在不遇到算术运算的情况下不会自动拆箱
  • 包装类的equals()方法不处理数据转型的关系

实战：插入式注解处理器

1.目标

实现检查Java程序是否符合命名规范.

  • 类（或接口）：符合驼式命名法，首字母大写。
  • 方法：符合驼式命名法，首字母小写。
  • 字段：
    • 类或实例变量。符合驼式命名法，首字母小写。
    • 常量。要求全部由大写字母或下划线构成，并且第一个字符不能是下划线。

2.代码实现

通过注解处理器API实现一个编译器插件.

  1. 继承抽象类javax.annotation.processing.AbstractProcessor

  2. 实现process()方法. 该方法是Javac编译器在执行注解处理器代码时要调用的过程.
    方法参数:
    annotations: 中获取到此注解处理器所要处理的注解集合
    roundEnv: 访问到当前这个轮次（Round）中的抽象语法树节点，每个语法树节点在这里都表示为一个Element。

    Java代码中可能出现的全部元素:
    包（PACKAGE）、
    枚举（ENUM）、
    类（CLASS）、
    注解（ANNOTATION_TYPE）、
    接口（INTERFACE）、
    枚举值（ENUM_CONSTANT）、
    字段（FIELD）、
    参数（PARAMETER）、
    本地变量（LOCAL_VARIABLE）、
    异常（EXCEPTION_PARAMETER）、
    方法（METHOD）、
    构造函数（CONSTRUCTOR）、
    静态语句块（STATIC_INIT，即static{}块）、
    实例语句块（INSTANCE_INIT，即{}块）、
    参数化类型（TYPE_PARAMETER，泛型尖括号内的类型）、
    资源变量（RESOURCE_VARIABLE，try-resource中定义的变量）、
    模块（MODULE）和未定义的其他语法树节点（OTHER）.

  3. AbstractProcessor中的一个protected变量: processingEnv
    它在注解处理器初始化的时候（init()方法执行的时候）创建，继承了AbstractProcessor的注解处理器代码可以直接访问它。它代表了注解处理器框架提供的一个上下文环境，要创建新的代码、向编译器输出信息、获取其他工具类等都需要用到这个实例变量。

  4. 添加注解@SupportedAnnotationTypes和@SupportedSourceVersion
    前者代表了这个注解处理器对哪些注解感兴趣，可以使用星号“”作为通配符代表对所有的注解都感兴趣，后者指出这个注解处理器可以处理*哪些版本的Java代码。

  5. 注解处理器在运行时是单例的. 若不需要改变或添加抽象语法树中的内容,process()返回false.
